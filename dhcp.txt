But "will they get re-scheduled automatically" depends on your config, so it's worth being deliberate.
They're separate schedulers
L3 agents schedule routers; DHCP agents schedule networks. Moving a router from one L3 agent to another (l3-agent-router-remove / l3-agent-router-add, or just letting the scheduler do it when you disable the old agent) has no effect on DHCP namespace placement. The qdhcp-* namespaces stay wherever the DHCP agent is currently hosting them.
Whether DHCP reschedules on its own
Check neutron.conf on your controllers:

dhcp_agents_per_network — if this is ≥ 2 (common in HA setups), each network already has redundant DHCP agents, so losing one is a non-event.
network_auto_schedule = True — newly created networks get auto-assigned to DHCP agents.
allow_automatic_dhcp_failover = True (default True) — when a DHCP agent goes dead (misses heartbeats past agent_down_time), Neutron will reschedule its networks to other live DHCP agents.

The catch: failover triggers on the agent being dead, not administratively disabled. If you openstack network agent set --disable <dhcp-agent> to drain it gracefully, networks bound to it won't auto-migrate — you have to move them yourself, or stop the agent so it goes dead.
Moving DHCP assignments manually
# See which networks a DHCP agent hosts
openstack network agent show <dhcp-agent-id>
openstack network list --agent <dhcp-agent-id>

# Remove and re-add
openstack network remove dhcp agent <old-dhcp-agent-id> <network-id>
openstack network add dhcp agent <new-dhcp-agent-id> <network-id>
The new agent will spin up a qdhcp-<network-id> namespace and dnsmasq within a few seconds; the old one tears its namespace down on removal.
Practical pattern for an agent-migration maintenance
If you're evacuating a network node entirely (both L3 and DHCP agents on it):

Disable the L3 agent → routers reschedule (assuming allow_automatic_l3_agent_failover = True) or move them manually with l3-agent-router-remove/add.
Move DHCP-hosted networks off explicitly with the remove/add commands above, or stop the DHCP agent service so it goes dead and let auto-failover handle it.
Verify with openstack network agent list that nothing is still bound to the old host before shutting it down.

Brief DHCP outage during a move is usually harmless (existing leases keep working), but if you've got short lease times or PXE-booting instances mid-migration, the explicit remove/add is cleaner than waiting for the dead-agent timer.
